## Schema design (H2)

### Core tables

* **`ppd_import_batch`**: tracks each dataset load (e.g., year 2024 file, imported_at, filename).
* **`ppd_transaction`**: typed and indexed “fact table” holding each transaction row.
* **`ppd_transaction_staging`**: all-`VARCHAR` landing table to make CSV loading easy (then transform into `ppd_transaction`).

### Reference/lookup tables (enforced by FK)

* `ref_property_type` (`D/S/T/F/O`)
* `ref_old_new` (`Y/N`)
* `ref_duration` (`F/L/...`)
* `ref_ppd_category` (`A/B`)
* `ref_record_status` (`A/C/D`) *(mostly for monthly files; yearly files often don’t include it)* ([GOV.UK][1])

### `V1__ppd_schema.sql`

```sql
-- =========================
-- HM Land Registry PPD schema (H2)
-- =========================

-- -------------------------
-- Reference tables
-- -------------------------
CREATE TABLE ref_property_type (
  code CHAR(1) PRIMARY KEY,
  description VARCHAR(50) NOT NULL
);

CREATE TABLE ref_old_new (
  code CHAR(1) PRIMARY KEY,
  description VARCHAR(50) NOT NULL
);

CREATE TABLE ref_duration (
  code CHAR(1) PRIMARY KEY,
  description VARCHAR(80) NOT NULL
);

CREATE TABLE ref_ppd_category (
  code CHAR(1) PRIMARY KEY,
  description VARCHAR(120) NOT NULL
);

CREATE TABLE ref_record_status (
  code CHAR(1) PRIMARY KEY,
  description VARCHAR(80) NOT NULL
);

-- -------------------------
-- Import batches (provenance)
-- -------------------------
CREATE TABLE ppd_import_batch (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  source_year INT, -- for yearly files; nullable if you ever load the full file
  source_file VARCHAR(255),
  imported_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  notes VARCHAR(1000)
);

-- -------------------------
-- Staging table (all strings; easiest for CSV bulk load)
-- -------------------------
CREATE TABLE ppd_transaction_staging (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  import_batch_id BIGINT NOT NULL,
  transaction_uuid_raw VARCHAR(80),
  price_raw VARCHAR(30),
  transfer_date_raw VARCHAR(40),
  postcode_raw VARCHAR(20),
  property_type_raw VARCHAR(5),
  old_new_raw VARCHAR(5),
  duration_raw VARCHAR(5),
  paon_raw VARCHAR(200),
  saon_raw VARCHAR(200),
  street_raw VARCHAR(200),
  locality_raw VARCHAR(200),
  town_city_raw VARCHAR(200),
  district_raw VARCHAR(200),
  county_raw VARCHAR(200),
  ppd_category_raw VARCHAR(5),
  record_status_raw VARCHAR(5), -- usually not present in yearly files (nullable)
  loaded_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_ppd_stage_batch
    FOREIGN KEY (import_batch_id) REFERENCES ppd_import_batch(id)
);

-- -------------------------
-- Typed transactions table
-- -------------------------
CREATE TABLE ppd_transaction (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,

  -- PPD "Transaction unique identifier" (UUID). Note: source often includes braces { }.
  transaction_uuid UUID NOT NULL,

  import_batch_id BIGINT NOT NULL,

  price INT NOT NULL CHECK (price > 0),
  transfer_date DATE NOT NULL,

  -- keep original postcode; also store a normalized version for indexing/search (upper, no spaces)
  postcode VARCHAR(10),
  postcode_norm VARCHAR(10),

  property_type_code CHAR(1) NOT NULL,
  old_new_code CHAR(1),
  duration_code CHAR(1),

  paon VARCHAR(200),
  saon VARCHAR(200),
  street VARCHAR(200),
  locality VARCHAR(200),
  town_city VARCHAR(200),
  district VARCHAR(200),
  county VARCHAR(200),

  ppd_category_code CHAR(1) NOT NULL,
  record_status_code CHAR(1), -- nullable for yearly files

  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

  CONSTRAINT uq_ppd_transaction_uuid UNIQUE (transaction_uuid),

  CONSTRAINT fk_ppd_txn_batch
    FOREIGN KEY (import_batch_id) REFERENCES ppd_import_batch(id),

  CONSTRAINT fk_ppd_txn_property_type
    FOREIGN KEY (property_type_code) REFERENCES ref_property_type(code),

  CONSTRAINT fk_ppd_txn_old_new
    FOREIGN KEY (old_new_code) REFERENCES ref_old_new(code),

  CONSTRAINT fk_ppd_txn_duration
    FOREIGN KEY (duration_code) REFERENCES ref_duration(code),

  CONSTRAINT fk_ppd_txn_category
    FOREIGN KEY (ppd_category_code) REFERENCES ref_ppd_category(code),

  CONSTRAINT fk_ppd_txn_record_status
    FOREIGN KEY (record_status_code) REFERENCES ref_record_status(code)
);

-- Helpful indexes for common API queries
CREATE INDEX idx_ppd_txn_transfer_date ON ppd_transaction(transfer_date);
CREATE INDEX idx_ppd_txn_postcode_norm ON ppd_transaction(postcode_norm);
CREATE INDEX idx_ppd_txn_district ON ppd_transaction(district);
CREATE INDEX idx_ppd_txn_town_city ON ppd_transaction(town_city);
CREATE INDEX idx_ppd_txn_county ON ppd_transaction(county);
CREATE INDEX idx_ppd_txn_price ON ppd_transaction(price);
CREATE INDEX idx_ppd_txn_type_date ON ppd_transaction(property_type_code, transfer_date);
```

## Notes you’ll need when importing the yearly CSVs

1. **UUID cleaning**: the dataset’s transaction id is often like `{75520218-C2D9-4926-95E3-00061A7A3784}` (braces). You’ll want to strip `{}` before inserting into `ppd_transaction.transaction_uuid`. ([GOV.UK][1])
2. **Postcode normalization**: set `postcode_norm = UPPER(REPLACE(postcode, ' ', ''))` so your API can search reliably.
3. **Record status**: for *yearly files*, `record_status_code` is typically **NULL** (it’s mainly described for monthly delta files). ([GOV.UK][1])

[1]: https://www.gov.uk/guidance/about-the-price-paid-data "How to access HM Land Registry Price Paid Data - GOV.UK"

## `V3__auth_schema.sql`

```sql
-- =========================
-- Authentication / Authorization tables
-- (simple RBAC: users <-> roles)
-- =========================

CREATE TABLE app_user (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,

  username VARCHAR(100) NOT NULL,
  email VARCHAR(255),
  password_hash VARCHAR(255) NOT NULL, -- store BCrypt hash (Spring Security default)

  enabled BOOLEAN NOT NULL DEFAULT TRUE,
  locked  BOOLEAN NOT NULL DEFAULT FALSE,

  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP
);

-- Unique constraints (H2 supports unique constraints)
ALTER TABLE app_user
  ADD CONSTRAINT uq_app_user_username UNIQUE (username);

-- Optional: if you want unique emails too
ALTER TABLE app_user
  ADD CONSTRAINT uq_app_user_email UNIQUE (email);


CREATE TABLE app_role (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name VARCHAR(50) NOT NULL -- e.g. ROLE_ADMIN, ROLE_USER
);

ALTER TABLE app_role
  ADD CONSTRAINT uq_app_role_name UNIQUE (name);


-- Many-to-many link table
CREATE TABLE app_user_role (
  user_id BIGINT NOT NULL,
  role_id BIGINT NOT NULL,
  assigned_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

  PRIMARY KEY (user_id, role_id),

  CONSTRAINT fk_user_role_user
    FOREIGN KEY (user_id) REFERENCES app_user(id)
    ON DELETE CASCADE,

  CONSTRAINT fk_user_role_role
    FOREIGN KEY (role_id) REFERENCES app_role(id)
    ON DELETE CASCADE
);

CREATE INDEX idx_app_user_role_user ON app_user_role(user_id);
CREATE INDEX idx_app_user_role_role ON app_user_role(role_id);
```

## `V4__link_imports_to_user.sql`

```sql
-- =========================
-- Link import batches to the user that performed the import
-- =========================

ALTER TABLE ppd_import_batch
  ADD COLUMN created_by_user_id BIGINT;

ALTER TABLE ppd_import_batch
  ADD CONSTRAINT fk_ppd_import_batch_created_by
    FOREIGN KEY (created_by_user_id) REFERENCES app_user(id);
```

This gives you clean auditability without forcing every transaction row to store a user.

### Notes for Spring Security compatibility

* Keep role names in the `ROLE_X` format; Spring’s defaults play nicely with that.
* Store **BCrypt** hashes in `password_hash` (length 60-ish, but `VARCHAR(255)` is safe).
* If later you add OAuth2/OIDC (Auth0/Keycloak), you can extend `app_user` with fields like `external_subject`, `provider`, etc., without changing the role model.
